CSR in the Go Engine

Compressed Sparse Row format for batched group topology — aligned with board_rules_checker.py

1. What is CSR?

The Problem

A Go board has groups — connected sets of same-colour stones. Each group can have a different number of stones. We need to store all groups across a whole batch of boards in one flat structure that the GPU can process in parallel.

The Solution: CSR

Compressed Sparse Row packs variable-length groups into two arrays: a flat data array of stone point ids, and a pointer array that says where each group starts and ends. No padding, no wasted memory.

CSR in one picture

Suppose we have 3 groups across a batch. Group 0 has stones at points [2, 7], group 1 at [5, 10, 11], group 2 at [0].

This is the standard CSR (Compressed Sparse Row) pattern from sparse matrix libraries, repurposed here for stone groups. The pointer array has length R+1 (one more than the number of groups), so that pointer[g+1] - pointer[g] gives the size of group g.

2. The Two-Layer ID Mapping

The engine manages groups at two scales simultaneously. This is the core design insight — and the part that can be confusing at first.

Layer 1: Board-Local Group IDs

Each board in the batch has its own 0-based group numbering. Board 0 might have groups {0, 1, 2}, board 1 might have groups {0, 1}. These are compact and independent per board.

Layer 2: Batch-Global Group IDs

Across the whole batch, every group gets a unique ID by adding a per-board offset. Board 0's groups stay at {0, 1, 2}, but board 1's groups become {3, 4}. These global IDs index directly into the CSR pointer array.

4. Mapping to board_rules_checker.py

The GroupCsr dataclass (line 68)

FieldShapeLayerRole
stone_point_ids (K,) int32 point Flat array of board-point coordinates for every stone, packed group-by-group across the whole batch
stone_pointer_by_group (R+1,) int32 global CSR indptr — indexed by batch-global group id to get the stone range
group_offset_by_board (B+1,) int32 local → global Cumsum of groups-per-board. Converts local group id to global: global = local + offset[b]
group_local_id_by_root_point (B, N2) int32 point → local LUT: union-find root point → 0-based local group id. -1 where no group root exists.

5. Why Two Layers?

ConcernLocal IDs aloneGlobal IDs aloneTwo layers (actual design)
Per-board reasoning Natural — 0-based per board Need to subtract offset every time Local for per-board logic, global for CSR
Batch-wide CSR indexing Can't index a shared array Natural — one flat namespace Global for shared CSR arrays
Capture metadata size Small (0..groups_in_board) Large (0..total_groups_in_batch) Store local (small), lift to global only when needed
Conversion cost — — One addition: local + offset[b]
