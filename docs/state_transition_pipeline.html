Analyze Once, Commit Once

How a bot queries legal moves and places a stone — the cache, state, and workspace lifecycle inside the Go engine

1. The Caller’s Contract — Two Calls per Turn

From any bot’s perspective the engine exposes exactly two instructions per turn. Everything else — union-find, CSR topology, Zobrist hashing, capture resolution — is machinery hidden behind these two calls.

1. Query — what can I play?

legal_points() → (B, N2) bool
Returns a mask of every legal board placement for the current side to move. Pass is always legal and handled at the action level — it is not in this mask.

Side effect: the engine analyzes the full position and caches the result. This is the expensive call — everything downstream rides for free on this analysis.

2. Commit — apply my chosen move

state_transition(action_ids) → GameState
Accepts (B,) action IDs (0..N2-1 = placement, N2 = pass). Mutates the board, pass counter, Zobrist hashes, and side-to-move in-place.

Key contract: the engine does not re-check legality. It reuses the cached analysis from the query call. The caller is responsible for only submitting legal actions.

A concrete example — RandomBot.select_actions():

# agents/random_bot.py — the simplest possible caller legal_points = engine.legal_points() # (B, N2) bool ← triggers full analysis + cache # build a (B, N2+1) probability mask: board placements + pass legal_actions[:, :N2] = legal_points legal_actions[:, N2] = True # pass is always legal action_ids = multinomial(legal_actions) # (B,) ← random choice # caller loop hands action_ids to the engine: engine.state_transition(action_ids) # reuses cached analysis — no redundant work

An MCTS bot follows the same pattern: call legal_points() once at the root, run simulations, then call state_transition() with the chosen action.

2. The State Register File

All mutable game state lives in four tensor “registers” inside a GameState dataclass. The GameStateMachine holds direct references to these tensors — no cloning, no indirection. Every register has a batch dimension for B parallel games.

boards
(B, N2) int8
The stone grid flattened to N2 = board_size² points. Each cell is 8 bits: EMPTY=0 BLACK=1 WHITE=2.
to_play
(B,) int8
1 bit of information in an int8 lane: 0=Black, 1=White. Flipped via XOR 1 after each non-terminal action. Color derivation: to_play + 1 maps directly to Stone enum values.
pass_count
(B,) int8
A 2-bit saturating counter (values 0, 1, 2) stored in int8. Incremented on pass, clamped at 2. Reset to 0 on any placement. ≥ 2 = game over.
zobrist_hash
(B, 2) int32
A two-slot shift register: [:,0] = current hash, [:,1] = previous hash. On commit, current shifts into previous; the new hash is written from precomputed candidates.

A fresh game starts from create_empty_game_state(B, board_size, device): all boards EMPTY, to_play = Black, pass_count = 0, both hashes = 0.

Why these four are sufficient

There is no “move history” tensor, no group table, no liberty count in the state. All structural information (connected groups, liberties, CSR topology, capturable neighbors, candidate hashes) is derived on-the-fly from boards + to_play during the analysis pass, cached for the duration of the turn, then discarded. The only historical information the engine retains is the single previous Zobrist hash — just enough for simple ko detection.

3. The Analyze-Once Pattern

Legality checking for Go is expensive because it requires full board analysis: finding every connected group, counting their liberties, identifying which opponent groups can be captured, and building the topological data needed for fast capture execution. This analysis is position-specific — it depends entirely on the current board state and cannot be carried over once a stone is placed.

The core insight: both the query (legal_points) and the commit (state_transition) need the same analysis. So the engine computes it exactly once, caches it, and both calls consume the same cached result.

The gatekeeper: _get_or_compute_latest()

Every path that needs the analysis goes through this single function. If the cache is warm (same position), it returns immediately. If any slot is None, it triggers the full analysis pipeline and populates all three cache slots atomically.

# Inside GameStateMachine def _get_or_compute_latest(self): if self._latest_legal_points_raw is None \ or self._latest_legal_info is None \ or self._latest_candidate_hashes is None: return self._compute_legal_and_candidates() # full analysis return ( # cache hit self._latest_legal_points_raw, self._latest_legal_info, self._latest_candidate_hashes, )

Cache lifecycle within one turn

Cache SlotShapePopulated byConsumed by
_latest_legal_points_raw (B, N2) bool _compute_legal_and_candidates legal_points — feeds into ko filter
_latest_legal_info LegalInfo _compute_legal_and_candidates state_transition — CSR metadata drives capture execution
_latest_candidate_hashes (B, N2) int32 _build_candidate_hashes legal_points (ko filter) + state_transition (Zobrist update)

After state_transition() mutates the board, it calls _invalidate_latest() which sets all three cache slots to None. The next legal_points() call will trigger a fresh analysis on the new position.

4. What the Analysis Discovers

The analysis pass is a pipeline inside BoardRulesChecker + GameStateMachine. Each stage builds on the previous one. Here is what gets computed, why it matters, and how the commit phase reuses it.

Neighbor Colors
_get_neighbor_colors_batch
For every point on every board, gather the colors of its 4 neighbors (N/S/W/E). Off-board edges are marked with a sentinel. This (B, N2, 4) int8 tensor is the foundation for everything that follows.
Union-Find
_batch_init_union_find
Batched hook-and-compress union-find groups same-colored adjacent stones into connected components. Produces two (B, N2) int32 tensors: roots (each stone’s group representative) and root_libs (liberty count per group root). Liberty counting uses sort + unique_consecutive to deduplicate shared liberties.
CSR Topology
_build_group_csr
Packs all stone groups across the entire batch into a single Compressed Sparse Row structure. This enables O(1) lookup of any group’s stones by global group ID — critical for fast capture execution.
stone_point_ids (K,) — all stone positions, group-major
stone_pointer_by_group (R+1,) — CSR row pointers
group_offset_by_board (B+1,) — local → global group ID offset
group_local_id_by_root_point (B, N2) — root → local group ID
Legality Check
compute_batch_legal_and_info
For every empty point, tests three conditions. A point is legal if any condition holds:
A) Has an adjacent empty point (immediate liberty after placement)
B) An adjacent opponent group has exactly 1 liberty (placement captures it)
C) An adjacent friendly group has >1 liberty (safe attachment, no suicide)
Also produces captured_group_local_ids (B, N2, 4) — for each point, the up-to-4 opponent groups that would be captured if a stone were placed there. This is what makes capture execution instant at commit time.
Candidate Hashes
_build_candidate_hashes
For every board point, computes the Zobrist hash that would result from placing a stone there. This is a multi-step XOR pipeline:
1. Placement delta — Z(empty) ⊕ Z(new_color) per point
2. Capture delta — prefix-XOR scan over CSR gives per-group hash contribution; up to 4 neighbor captures XOR’d per point
3. Assemble — current_hash ⊕ placement_delta ⊕ capture_delta
These precomputed hashes serve double duty: ko filtering (compare against previous hash) and Zobrist update at commit time (direct lookup by chosen point).

5. The Commit Pass — Consuming the Analysis

When the bot calls state_transition(action_ids), the engine consumes the cached analysis through a lightweight sequence of in-place mutations. No union-find, no CSR build, no Zobrist recomputation — all of that was done during the query.

Normalize Actions
Classify each action: placement vs. pass vs. terminal no-op. Finished games (pass_count ≥ 2) are force-routed to the pass path regardless of the action submitted.
Reads: action_ids pass_count
Produces: point_ids is_pass play_mask finished
Update pass_count
The 2-bit counter shifts: placement → reset to 0, pass → increment (clamped at 2).
Mutates: pass_count
Capture + Place
For each active game with a placement, uses the cached LegalInfo to resolve captures:
1. Look up captured_group_local_ids[b, point] → up to 4 group IDs
2. Convert local → global group ID via group_offset_by_board
3. Dereference CSR: stone_pointer_by_group[g]..stone_pointer_by_group[g+1] gives the captured stones
4. Set captured stones to EMPTY, then write new stone
Mutates: boards — Reads from cache: LegalInfo.csr + captured_group_local_ids
Zobrist Shift
The hash shift register advances: prev ← cur, then for placement games: cur ← candidate_hashes[b, point]. The candidate hash was precomputed during analysis — no XOR pipeline runs here.
Mutates: zobrist_hash — Reads from cache: candidate_hashes
Side Flip +
Cache Flush
to_play ^= 1 for non-finished games (a single XOR flips Black↔White). Then _invalidate_latest() sets all cache slots to None, ensuring the next legal_points() triggers a fresh analysis on the new position.
Mutates: to_play — Clears: all cache slots

6. Workspace Reuse — Persistent Buffers

Tensor allocation on GPU is not free. The engine uses a capacity-based lazy-alloc pattern: scratch buffers are allocated once on first use, then reused across every subsequent call without reallocation. This mirrors how a CPU reuses register files and scratchpad memory across instructions.

BoardRulesChecker workspaces
_uf_neighbor_parent — (B, N2, 4) int32 — union-find scratch for pointer jumping
_stone_point_ids_buf, _stone_pointer_buf, _group_offset_buf, _group_local_id_buf — CSR output buffers, capacity-tracked
point_ids_i32 — precomputed arange(N2) reused as initial UF parent array
GameStateMachine workspaces
_placement_delta_buf — (B, N2) int32
_capture_delta_buf — (B, N2) int32
_candidate_hashes_buf — (B, N2) int32
_cap_vals_buf — (B, N2, 4) int32
_group_xor_buf — (B*N2,) int32
_prefix_scratch_buf — (B*N2,) int32 — eliminates per-iteration clones in prefix-XOR scan

All workspace buffers are allocated with torch.empty (no zeroing cost) and only grow if the batch × board size increases. Between calls, the tensors stay resident on the GPU — only the data inside them changes.

Appendix: Why Analyze-Once Matters for Go

The position-specific structure problem

Legality checking in Go is not a simple rule lookup. It requires discovering the full topological structure of the board — which stones are connected, how many liberties each group has, which groups are capturable, and what the board hash would look like after each possible move. This structural analysis is:

  • Expensive — union-find, sorting, CSR construction, prefix-XOR scans
  • Position-specific — every placement changes the group structure, invalidating everything
  • Shared — the legality query and the move commit need the exact same structural data

The analyze-once pattern exploits this: compute the full structural analysis when the bot asks “what can I play?”, cache it, and when the bot says “play here”, consume the cached result to apply captures and update hashes in a single pass. The commit is essentially a table lookup into the analysis — index the CSR by the chosen point’s capture IDs, index the candidate hashes by the chosen point, done.
